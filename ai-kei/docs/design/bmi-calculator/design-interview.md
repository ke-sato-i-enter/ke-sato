## ヒアリング目的

既存の要件定義・技術スタック・ノートを確認し、軽量設計に必要な最小限の設計判断を確定するためにヒアリングを実施しました。

## 質問と回答

### Q1: 基本アーキテクチャ方針を選んでください

**質問日時**: 2026-06-05
**カテゴリ**: アーキテクチャ
**背景**: 軽量設計では、将来拡張前提で分離設計にするか、現行のクライアント完結型を維持するかで文書粒度が変わるため。

**回答**: バックエンドレスSPA維持

**信頼性への影響**:
- この回答により、アーキテクチャパターンの信頼性レベルが 🟡 → 🔵 に向上
- API/DB設計を軽量設計のスコープ外として確定できた

---

### Q2: 設計で明示する主要技術を選択してください

**質問日時**: 2026-06-05
**カテゴリ**: 技術選択
**背景**: 要件上は複数実装方式が取り得るため、設計で固定すべき技術要素を明確化する必要があるため。

**回答**: React Hooks中心 / Tailwind CSS / LocalStorage履歴管理

**信頼性への影響**:
- この回答により、状態管理と永続化戦略の信頼性レベルが 🟡 → 🔵 に向上
- dataflow.md の状態遷移と履歴フローを確定できた

---

### Q3: 最低限のデータモデル方針を選んでください

**質問日時**: 2026-06-05
**カテゴリ**: データモデル
**背景**: LocalStorage保存対象を履歴限定にするか、入力状態まで含めるかでデータ整合性設計が変わるため。

**回答**: 計算履歴のみ

**信頼性への影響**:
- この回答により、保存スキーマ定義の信頼性レベルが 🟡 → 🔵 に向上
- 履歴データ項目を heightCm, weightKg, bmi, standardWeightKg, diffKg, calculatedAt に固定できた

---

### Q4: Phase 1で優先する項目を選択してください

**質問日時**: 2026-06-05
**カテゴリ**: パフォーマンス
**背景**: MVPの軽量設計では優先機能を先に確定し、不要な設計拡張を避ける必要があるため。

**回答**: 入力バリデーション / BMI・標準体重・差分の計算表示 / BMI基準解説+計算式トグル

**信頼性への影響**:
- この回答により、Phase 1範囲の機能設計が 🔵 として固定
- 履歴表示高度化などの後続項目を残課題として切り分け可能になった

## ヒアリング結果サマリー

### 確認できた事項
- バックエンドレスSPAを維持する
- 主要技術は React Hooks/Tailwind/LocalStorage とする
- データモデルは計算履歴に限定する
- Phase 1優先は入力検証・計算表示・解説表示とする

### 設計方針の決定事項
- 軽量設計として architecture.md と dataflow.md を中心に作成する
- API仕様、DBスキーマ、型定義は本フェーズでは作成しない

### 残課題
- LocalStorage無効環境時のUX文言（通知方法）の最終決定
- 履歴機能の表示UI詳細（一覧件数、削除操作）の次フェーズ定義
